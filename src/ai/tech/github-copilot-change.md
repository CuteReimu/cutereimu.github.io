2026年6月1号起，Github Copilot 将会调整定价模式，这个在 [Github 的官方 Blog](https://github.blog/news-insights/company-news/github-copilot-is-moving-to-usage-based-billing/) 中也有提及。曾经，它是按照高级请求的次数来计费的，因此我们可以把多个问题放在同一个请求中进行，以此来节约成本。但现在，随着 Github Copilot 调整为基于使用量的计费模式后，这种做法就不再适用了。

关于这个事情，其实也在很多开发者的预期当中，毕竟曾经的那种计费模式在如今的环境中确实已经不太合适了。现在的 AI，例如 Claude Code、 Codex 等，功能十分强大，但收费也不菲。而在曾经的按次计费模式下，开发者们可以通过较低的花销，来让 Github Copilot 使用上述的这些模型，达到相同的效果，我猜这背后 Github Copilot 也做了不少赔钱的买卖。但这种模式显然是不可持续的，因此调整为基于使用量的计费模式也是情理之中的事情了。

## 影响有多大？

这应该是大家非常关心的一个问题了，现在 Github 给了一个预览数据，可以看到4月的使用量改为按量计算后的结果。我简单分享一下：

最多的一天，我使用了27次高级请求，按照之前的按次计费，每次花费 \$0.04 ，总共是 \$1.08 。但按照新的按量计费模式，27次高级请求的总花费是 1008 个 AI Cridits ，按照官方给出的每个 AI Cridits 1美分的定价来计算，就是 $10 了，翻了大概10倍左右。大部分日期都是这样的情况，翻了 7 ~ 30 倍；极少数日期因为只聊了一两句话，用新的计费方式会有所减少。从当月总量来看，总计翻了18倍，是一个非常恐怖的增长了。

可以看出，这对于重度 AI 使用者来说，是一个重大的打击。对于在大厂工作的开发者们，一般情况下公司会负担这部分成本，让你们能够更好地走在时代的前沿，是一个不错的机会。但对于没有这样福利的小企业或者个人开发者，这就是一个不小的负担了。

## 有没有什么替代方案？

还是得摆正一下心态，Github Copilot 这个操作，并不能算作“涨价”，应该说是把计费拉升到和同行相同的一个水平了。

这样说来，其实替代方案就很明确了，要么就接着使用 Github Copilot ，或者干脆使用 Claude Code、 Codex、 Cursor 等，要么就使用国内的类似工具，例如阿里的通义灵码、字节的豆包编程助手、或者DeepSeek‑Coder等。

## 展望

我们就以上面那个一天 \$10 的数据为例，或者更夸张一点，就算一天 \$50 ，这也明显低于一线城市的程序员的收入水平。而且公司还不用为AI交社保。随着 AI 的不断发展，这个成本会继续逐渐降低，能力会越来越出色。这意味着什么，想必大家都明白。所以真的很有必要规划自己的职业方向，要能做一些AI做不到的事情，才能保证自己未来的竞争力。
